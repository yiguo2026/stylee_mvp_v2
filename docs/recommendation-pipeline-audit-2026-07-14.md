# 穿搭推荐链路审计与改进记录（2026-07-14）

本文记录 2026-07-13～14 对真实线上推荐 case“去海岛度假穿什么”的代码审计结论。
文档描述的是当前可审计的程序路径，不代表模型不可见的内部思维链。

## 1. 已确认现象

- 一次真实 `model-service/deepseek` 推荐耗时 18.7 秒。
- 返回 1 件 T 恤、2 条牛仔短裤、1 双帆布鞋；重复下装属于硬约束失效。
- 推荐补位单品没有图片。
- “换一套看看”在只有一套结果时会重新调用模型；服务瞬时失败或返回空结果时，App 会展示本地 fallback。
- 模型可能返回“补：建议购买一件……”这类过长名称，不适合作为普通标签。

## 2. 当前执行路径

1. 首页检查登录和客户端本地额度，将文本、标签和当前城市天气放入结果页路由。
2. 结果页读取衣橱与风格偏好，构造 `n=3` 的推荐请求。
3. App 携带 Supabase access token 请求 Render `POST /recommend`。
4. model service 执行 CORS、Supabase token 验证、每用户分钟限流和请求体限制。
5. B0：DeepSeek 把自然语言解析为 `SceneSpec`。
6. B1：代码按季节、温度、槽位和衣橱状态构造候选池。
7. B2：优先向量检索 Garments2Look 范例；索引不可用时退到关键词 stub。
8. B3：DeepSeek 生成搭配草稿。App 请求 3 套，但 pipeline 默认过生成 6 套。
9. B4：代码做槽位硬校验，并按身材 35%、场景 30%、风格 20%、色彩 15% 打分。
10. B5/B6：去重、多样性排序、置信度计算。
11. adapter 转成 App JSON，前端关联已有衣橱图片并渲染推荐补位单品。

## 3. 根因与修复决策

### 3.1 18.7 秒

主要瓶颈是 B0 和 B3 两次串行 DeepSeek 调用，以及 B3 一次要求输出 6 套 JSON。
此外可能叠加 Render 冷启动和 Supabase token 首次远程校验。页面显示的 18.7 秒是服务请求计时，
不是 9 秒加载进度动画。

后续性能方案：常驻 Render、合并/代码化 B0、首屏只生成 1 套、缩小推荐专用输出 token、
加入阶段耗时 trace，并设置 8～9 秒 deadline 和确定性 fallback。稳定 10 秒目标必须按 p95 验证。

### 3.2 两条短裤

旧校验对推荐 gap 单品使用模型填写的 `role`，但页面按 `gap.category` 展示。
模型把第二条短裤写成 `role=accessory, category=下装` 时，校验认为它是配饰，页面却显示为下装。

修复决策：gap 槽位一律由固定 `CATEGORY_SLOT[category]` 推导；已有单品一律由真实衣橱数据推导；
推荐连衣裙也必须覆盖 torso + bottom；重复核心槽位必须被 B4 拒绝并由测试锁定。

### 3.3 “换一套”偶发 fallback

旧方案签名只包含 `owned_item_ids`。空衣橱时所有草稿签名都是空集合，6 套方案会被错误折叠为 1 套。
结果页发现只有一套后，每次点击“换一套”都会重新请求模型，因此瞬时服务错误会触发本地 fallback。

修复决策：签名同时包含已有单品和推荐 gap 的品类/名称，使多套推荐保留在前端并本地轮换；
若确实只有一套且重新请求失败，保留当前有效搭配并提示，不再用 fallback 覆盖屏幕。

### 3.4 推荐名称过长

旧 adapter 直接输出 `"补:" + gap.desc`，模型会把购买建议句当作标签名称。

修复决策：prompt 要求 `gap.desc` 只返回不超过 12 个汉字的单品名；服务端确定性压缩名称；
前端再做一次兼容清洗，以覆盖尚未升级的旧服务响应。

### 3.5 没有图片

App 的 `RecommendedItem` 和结果页已支持 `image_url`，但 model service 响应契约没有图片字段，
adapter 也没有目录匹配步骤。已有衣橱单品可显示原图；纯推荐 gap 只能显示类别图标。

建议在主 Supabase 建 `recommendation_catalog` 和 `catalog-images` Storage：DeepSeek 只产出结构化需求，
model service 按槽位/品类硬过滤并匹配 catalog item，返回稳定的 `catalog_item_id + image_url`。
图片生成只用于离线目录生产或异步整套 flatlay，不进入 10 秒首屏关键路径。

## 4. 当前页面排布

- 主区顺序：外套 → 上装 → 下装/连体装 → 鞋。
- 外套/上装在顶部横排，下装在中部纵排，鞋在底部横排。
- 帽巾、包袋、配饰在右侧纵排。
- “搭配单品”宫格先显示已有单品，再按后端顺序显示推荐单品。
- 前端不应承担业务纠错；核心槽位合法性由 model service 保证，前端只保留最后一道防御。

## 5. 仍待完成

- 推荐接口阶段耗时 trace 和 p50/p95 观测。
- 单次模型调用的快速路径和 10 秒 deadline。
- 主 Supabase 推荐目录及图片匹配。
- 主 Supabase 权限/每日额度的服务端权威校验；当前 App 每日额度仍是客户端 AsyncStorage。
