# Stylee AI entitlement and quota architecture

Status: proposed for the next implementation phase. The initial Render service
is for authenticated smoke testing; it is not a production entitlement system.

## 1. Decision

User model capabilities, plan membership and quota balances are product data and
must use the Stylee App's primary Supabase as the source of truth. The optional
model-service usage Supabase remains an observability sink for detailed provider
tokens, latency and cost. It must never decide whether a request is allowed.

| Data | Authoritative location | Reason |
|---|---|---|
| User identity | Main Stylee Supabase Auth | Existing product identity |
| Plan/subscription | Main Stylee Supabase | Product/billing lifecycle |
| Capability permissions | Main Stylee Supabase | Must follow the user across devices/services |
| Daily/monthly balances | Main Stylee Supabase | Needs transactional, concurrent enforcement |
| Reservation/finalization ledger | Main Stylee Supabase | Required for atomic charging and refunds |
| Detailed tokens/model/latency/cost | Optional model usage Supabase | Operational analytics, not authorization |
| Per-minute abuse limit | In model service | Fast local protection independent of plan quota |

The current App `AsyncStorage` quota is retained only as a temporary display
cache. It is not trusted for authorization because it is device-local and can
be reset or raced.

## 2. Capability vocabulary

Every paid operation has a stable server-owned capability code:

```text
recognize
recognize_multi
standardize
recommend
intent
reason
product_extract
tryon_suggestion
tryon_image
```

The client requests a product capability, never a provider model. Plan policy
maps the capability to an allowed model tier, output-token cap and units. This
prevents a modified client from selecting an expensive model.

## 3. Main Supabase schema

Recommended private/product tables:

### `ai_plans`

`plan_code`, display name, status and effective dates. Examples: `free`, `pro`,
`internal_tester`.

### `ai_plan_capabilities`

One row per plan/capability:

- `plan_code`, `capability_code`
- `enabled`
- `window_type` (`day`, `month`, `lifetime`)
- `limit_units`
- `max_model_tier`
- `max_output_tokens`
- `max_concurrency`

### `user_ai_entitlements`

The user's active plan and validity window. Written only by trusted admin,
billing webhook or support tooling, never by the App.

### `user_ai_entitlement_overrides`

Optional time-bounded per-user capability/limit overrides for trials, support,
internal testing or promotions. Explicit overrides take precedence over plan
defaults and are auditable.

### `ai_quota_buckets`

Atomic counter per `(user_id, capability_code, window_start)` containing
`reserved_units` and `consumed_units`. A database transaction must reject a
reservation if `reserved + consumed + requested > effective_limit`.

### `ai_usage_reservations`

Immutable/auditable request ledger:

- `reservation_id`, `idempotency_key` (unique), `request_id`
- `user_id`, `capability_code`, reserved units
- `status` (`reserved`, `succeeded`, `failed`, `expired`)
- effective plan/model-policy snapshot
- actual tokens/images/cost summary
- creation, expiry and finalization timestamps

Reservations prevent concurrent requests from all seeing the same remaining
balance. Idempotency prevents retries from charging twice.

## 4. Enforcement flow

```text
App + user JWT
  -> model service verifies JWT and per-minute rate limit
  -> main Supabase quota Edge Function: reserve
       -> validates the same user JWT
       -> atomic SQL reserve transaction
       -> returns reservation_id, remaining, model policy
  -> model service enforces returned model tier/token cap
  -> provider call
  -> main Supabase quota Edge Function: finalize
       -> success: reserved -> consumed, record actual units
       -> provider failure: release reservation (or charge attempt policy)
  -> optional async detailed event to usage Supabase
```

The quota Edge Function belongs to the main product backend. It holds any
database/admin credential internally; the App and model service continue to use
only the user's JWT plus public Supabase project configuration. The underlying
reserve/finalize SQL RPC is callable only by that trusted function, not `anon`
or `authenticated` roles directly.

## 5. API contract

Reserve input (generated by model service, not trusted from the client):

```json
{
  "capability": "recommend",
  "requested_units": 1,
  "idempotency_key": "request-uuid"
}
```

Allowed response:

```json
{
  "allowed": true,
  "reservation_id": "uuid",
  "remaining": 4,
  "model_policy": {
    "tier": "flash",
    "max_output_tokens": 2048
  }
}
```

Denials use stable semantics:

- `403 capability_disabled`
- `429 quota_exhausted`
- `409 request_in_progress` for an idempotency collision
- `503 quota_service_unavailable`

Costly provider calls fail closed when quota authorization is unavailable. A
telemetry outage does not block calls because telemetry is not authoritative.

## 6. Security and privacy

- Main quota tables are not directly writable by `anon` or `authenticated`.
- Users may receive a read-only projection of their own effective capability and
  remaining balance for UI display.
- Plan changes are admin/billing events with an audit trail.
- Provider keys remain only in model service.
- Supabase service-role/secret remains only inside the main Supabase Edge
  Function environment, not in App/Web or model service.
- Detailed usage logs must not contain prompts, uploaded images, access tokens or
  provider keys; store only identifiers, model, units, latency, status and cost.
- The usage analytics project should remove anonymous read/write before it is
  used with production user identifiers.

## 7. Reconciliation and operations

- A scheduled job expires abandoned reservations and releases their units.
- Daily reconciliation compares finalized reservations with provider billing
  summaries and the optional detailed usage sink.
- Alert on reservation/finalization mismatch, abnormal cost per user, repeated
  provider failures and quota-denial spikes.
- Per-minute model-service rate limiting remains a separate abuse-control layer;
  it must not be presented as the user's plan balance.

## 8. Migration plan

1. Deploy the authenticated model service for internal smoke tests only.
2. Add main-Supabase entitlement tables, grants/RLS, atomic SQL functions and
   the quota Edge Function.
3. Add model-service reserve/finalize client with fail-closed production mode.
4. Replace App AsyncStorage quota authority with a read-only server balance;
   retain local state only as a short-lived UI cache.
5. Harden the usage analytics project and send detailed events only from model
   service.
6. Backfill initial plans/entitlements, run concurrency/idempotency/refund tests,
   then enable production traffic.
